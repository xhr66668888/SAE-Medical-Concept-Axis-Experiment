# 实验介绍：用 Gemma 3 与 Gemma Scope 2 分析医学概念表示

本实验尝试用开源大模型和稀疏自编码器（Sparse Autoencoder, SAE）观察语言模型在医学提示词中的内部表示。具体来说，我们关心 Gemma 3 在读到糖尿病诊断、ICD 编码、并发症描述和 ATC 药物上下文时，残差流中是否出现了可分离、可追踪的医学概念结构。

这个项目不是为了生成医疗建议，而是为了做机制可解释性实验：把模型内部激活取出来，用 SAE 分解成稀疏特征，再观察这些特征如何随疾病类型和并发症语境变化。

## 为什么做这个实验

大模型在医学文本中能表现出一定的诊断、编码和药物知识，但这些能力通常只从输出结果上观察。我们希望进一步问一个更内部的问题：模型在回答之前，是否已经在隐空间里区分了不同医学概念？

例如，Type 1 diabetes 和 Type 2 diabetes 在表面文本上很相近，但医学含义、治疗线索和常见药物不同。如果模型真的形成了稳定表示，那么中间层 residual stream 中可能存在一个方向，能把 Type 1 和 Type 2 的激活分开。这个方向就是本实验中的“医学概念轴”。

另一个问题是并发症语境。加入 kidney complications 或 neurological complications 后，模型是否会激活不同的 SAE 特征？这些特征能否从早期层到晚期层形成一条可视化的“思考路线”？这就是本实验中 circuit tracing 部分要观察的内容。

## 实验目的

本实验有三个主要目标：

1. 构造医学对比提示词，让 Type 1 / Type 2 diabetes、kidney / neurological complications 等变量可控变化。
2. 从 Gemma 3 的 residual stream 中提取激活，计算 Type 1 与 Type 2 的线性概念轴，并用 PCA 图观察是否可分。
3. 使用 Gemma Scope 2 SAEs 将激活分解为稀疏特征，筛选只在特定并发症下增强的 feature，并画出跨层特征路线图。

## 数据来源

本实验使用仓库中的两个结构化医学编码文件：

- `data/icd_diagnosis_ccs.csv`：ICD-9 / ICD-10 诊断编码及 CCS 分类信息。
- `data/atc_drug_hierarchy.csv`：ATC 药物编码及药物层级信息。

生成脚本会从 ICD 文件中抽取 Type 1 diabetes 与 Type 2 diabetes 相关诊断，例如：

- `E10.22`: Type 1 diabetes mellitus with diabetic chronic kidney disease
- `E11.22`: Type 2 diabetes mellitus with diabetic chronic kidney disease
- `E10.40`: Type 1 diabetes mellitus with diabetic neuropathy
- `E11.40`: Type 2 diabetes mellitus with diabetic neuropathy

同时从 ATC 文件中保留糖尿病相关药物提示，例如 insulin 和 metformin。最终生成的 prompt 会写入：

```text
data/diabetes_contrastive_prompts.csv
```

## 模型与 SAE 选择

当前机器没有 CUDA，但有 Intel Lunar Lake Arc Graphics 130V / 140V 集成 GPU，因此默认实验路线改为 Intel XPU 优先：

- 默认模型：`google/gemma-3-1b-it`
- 默认 SAE：`gemma-scope-2-1b-it-res`
- 默认设备：`xpu`
- 默认精度：`bfloat16`

这条路线是真正的 Gemma 3 + Gemma Scope 2 组合，并且比较适合当前约 15GB 统一内存的硬件。

项目还保留两个备用路线：

- `gemma3-270m-it-res`：最小 Gemma Scope 2 路线，适合 CPU 或快速冒烟测试。
- `gemma3-4b-it-res`：本机可尝试的上限，但需要减少 prompt 数量和层数。

不建议在当前机器上本地跑 Gemma 3 12B / 27B 配合 TransformerLens 和 SAEs，因为模型权重、激活缓存和 SAE 权重叠加后会明显超过可用内存。

## 实验流程

### Step 1：MVP 激活提取

第一步只验证最小链路是否能跑通：

1. 加载 Gemma 3 模型。
2. 加载单层 residual stream SAE。
3. 输入两条 hardcoded diabetes prompts。
4. 用 TransformerLens 截获指定层 residual stream。
5. 将 residual activation 输入 SAE。
6. 打印最活跃的 Top-K feature IDs。

这一步的价值是尽早发现环境、权限、模型加载、hook 名称和 tensor 维度问题。

### Step 2：生成对比 Prompts

第二步从 ICD/ATC CSV 自动生成对比提示词。变量包括：

- 疾病类型：`type1`, `type2`
- 并发症：`none`, `kidney`, `neurological`, `ophthalmic`, `circulatory`
- 模板变体：默认每个组合 3 条

这些 prompt 的核心设计是只改变少量医学变量，让后续激活差异更容易解释。

### Step 3：医学概念轴

第三步计算 Type 1 vs Type 2 diabetes 的概念轴：

```text
Axis_Type = mean(Type 1 activations) - mean(Type 2 activations)
```

脚本会在指定层和指定 token 位置提取 residual activation，并输出：

- `concept_axis.pt`：概念轴和类别均值向量。
- `axis_results.csv`：每条 prompt 的 PCA 坐标、axis projection 和预测类别。
- `concept_axis_pca.png`：Type 1 / Type 2 激活的二维 PCA 散点图。
- `axis_summary.txt`：线性切分准确率和 PCA 解释方差。

如果 Type 1 和 Type 2 在 PCA 图中呈现分离，且沿概念轴投影可区分，就说明模型内部表示中存在可测的疾病类型方向。

### Step 4：跨层 SAE 特征路线

第四步观察并发症语境带来的差异特征。例如比较：

```text
type1 + kidney complications
vs
type1 + neurological complications
```

每一层会做：

1. 提取 target prompts 的 SAE feature activations。
2. 提取 baseline prompts 的 SAE feature activations。
3. 计算 target - baseline 的差异。
4. 过滤两组中都高频激活的通用语法特征。
5. 选出 top differential features。

跨层图中的节点是不同层的差异 feature，边使用 SAE decoder direction 的余弦相似度作为启发式连接强度。输出包括：

- `circuit_route.txt`：纯文本特征路线。
- `circuit_features.csv`：每层 top features。
- `circuit_edges.csv`：跨层连接边。
- `circuit_graph.png`：networkx 可视化路线图。
- `circuit_graph.graphml`：可导入 Gephi / Cytoscape 的图文件。

需要注意的是，这里的边是相似度启发式连接，不等同于严格因果路径。若要证明因果关系，还需要后续加入 activation patching、feature ablation 或 causal intervention。

## 预期观察

如果实验顺利，可能观察到：

- Type 1 和 Type 2 diabetes prompts 在中间层 residual stream 中沿某个方向分离。
- kidney complications 相比 neurological complications 激活不同的 SAE features。
- 早期层更偏词面和编码信息，中间层更偏疾病/并发症语义，后期层更接近药物或回答投射。

这些观察不保证一定出现，因为 SAE feature 的解释性、模型医学知识、prompt 模板和目标 token 位置都会影响结果。但即使结果不明显，输出的激活、PCA 和 feature 表也能帮助继续调整实验设计。

## 实验边界

本实验有几个重要限制：

- SAE feature ID 本身不自动等于人类可读概念，需要结合 Neuronpedia、top activating examples 或人工解释进一步命名。
- PCA 图只能提供低维观察，不能证明模型真的“理解”医学概念。
- Step 4 的图是启发式 feature route，不是严格 causal graph。
- 当前硬件适合 270M / 1B / 小规模 4B 实验，不适合大规模 Gemma 3 12B / 27B 分析。
- 输出不能用于临床决策，只用于机制可解释性研究。

## 如何阅读结果

建议按以下顺序看输出：

1. 先看 Step 1 的 top feature IDs，确认 SAE 激活非空且 shape 正常。
2. 再看 `data/diabetes_contrastive_prompts.csv`，确认 prompt 是否符合实验变量设计。
3. 看 `outputs/axis/concept_axis_pca.png`，观察 Type 1 / Type 2 是否分开。
4. 看 `outputs/axis/axis_summary.txt`，确认概念轴投影分类效果。
5. 看 `outputs/circuit/circuit_route.txt`，快速理解每层 top 差异 feature。
6. 看 `outputs/circuit/circuit_graph.png` 或 `circuit_graph.graphml`，观察跨层 feature route。

这个实验的核心价值不是一次性得出最终医学结论，而是建立一条可复现的分析管线：从医学编码数据到 prompt，从 prompt 到模型激活，从激活到 SAE feature，再从 feature 到可视化解释。
